# agent-image：对话式图像 Agent（需求说明）

## Problem Frame

单机自用的**玩具级**产品：用 **Next.js 全栈壳 + 对话界面** 驱动「类 Cursor / Claude Code」的多轮 Agent 体验，**默认任务偏向图像生成**（及与生成强相关的编排；**专业「编辑」能力以 Scope 中 v1 范围为准**），但 Agent 本身保持通用能力（工具、可见的中间步骤）。用户通过**主观体验**判断是否好用；**不做**科学对照实验或严谨效果证明。

核心矛盾：多厂商 LLM / 生图 API 的 **baseURL、鉴权、参数形态、能力上限** 各不相同；系统必须支持**多条 Model 配置**，且**不能**依赖「整个应用只有一个环境变量 API Key」的模型。

**相关文档（推荐性，非硬性 R 条）：** Agent 默认工作流与能力告知要点见 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)，与本文 R8–R12, R14–R19 及 F3 对齐，供 Prompt / 实现对照。

---

## Actors

- A1. **终端用户（本人）**：配置 Model、在对话页选择当前会话使用的模型、与 Agent 多轮交互、查看工具与上下文用量。
- A2. **Agent（服务端编排的 LLM）**：在**用户已选定**的 LLM 与（可选的）主 / 次生图 Model 约束下，选择工具、产生可见推理 / 工具步骤；**不负责**切换厂商或替换用户选定的 Model；**也不负责**在 LLM 能力不足（如缺 Vision）时自动切换 Model（见 Dependencies「LLM Vision 假设」）。

## Key Flows

- F1. **配置 Model**
    - **Trigger：** 用户打开 Model / 设置相关界面。
    - **Actors：** A1
    - **Steps：** 新增或编辑一条 Model；填写**类型**（**LLM** / **生图** / **Search**——一条 Model 仅绑一种类型，见 Key Decisions「单条 Model = 单一类型」）；按类型分支填写字段：
        - **LLM**：厂商适配标识、**baseURL**、**API Key**、必要头、**上下文窗口上限**（用于 R6 用量可视化）、其他能力元数据（如是否支持 Vision，假定为是，见 Dependencies）。
        - **生图**：厂商适配标识、**baseURL**、**API Key**、必要头、**支持的输出分辨率 / 比例枚举**（来自模型 / 厂家真实能力）、其他能力元数据（如是否支持负向提示、参考图字段、最大参考图张数等，供工具 schema 与 R5 校验使用）。
        - **Search**：按厂商 schema 分支字段（如 Brave Search 不沿用 baseURL 概念，使用专属字段；见 R17）。
        
        **Search 类**按 **R17** 配置：可插拔多 Search Model，且 **web-search** 与 **image-search** 可绑定**不同** Model；**web-fetch** 不绑定 Search Model。
    - **Outcome：** 持久化存储中存在可用的 Model 记录；**不存在**「全局唯一 `API_KEY` 环境变量」作为产品前提；密钥按 **明文** 存储（玩具 / 单机自用，加密 deferred，见 Key Decisions「密钥持久化」）。
    - **Covered by：** R2, R3, R4, R10, R11, R17

- F2. **对话中生成与工具可见性**
    - **Trigger：** 用户在对话页发送消息；当前会话已选择 **LLM**（必选），可选已选择主生图、次生图（若适用，按 R3 / R9 暴露规则决定生图工具是否可用）。
    - **Actors：** A1, A2
    - **Steps：** 请求经服务端调用选定 LLM；模型可调用通用工具（如联网搜索、抓取、以图搜图等，具体清单可迭代）及 **主/次** 生图工具；工具执行结果与中间产物在对话中**可读**；生图前可有搜索、分析、改写等步骤，**不把生图当作单一黑盒一次调用**。
    - **Outcome：** 用户能看到 Agent 做了什么；对话中可出现多张图（检索、中间稿、成品等），且 Agent 能**引用本轮对话内已出现的图像**（存储与 URL 规则见 **R18**；多模态序列化细节可在规划中细化）。**用户可在任意时刻按 R19 中断当前轮次。**
    - **Covered by：** R1, R5, R6, R7, R8, R9, R14–R19

- F3. **Agent 收束、意图与失败恢复（产品行为）**
    - **Trigger：** 多轮对话中模型正在执行任务或工具返回结果/错误。
    - **Actors：** A1, A2
    - **Steps：** Agent 根据对用户消息的理解区分 **闲聊 / 讨论 / 动手执行 / 分步引导** 等意图并调整策略（**不要求**单独代码层意图模块，见 R14）。**生图类工具**：模型可产生拟执行生图的工具意图；**须先经 UI 确认，再调用真实生图 API**（见 R15）。**不设**确认超时；用户仅 **确认** 或 **拒绝**（二元，不区分「跳过」）。**拒绝** 作为该次工具调用的 **返回内容**（语义为「用户未批准本次工具调用」）回传给模型，**不**执行生图。**不要求**将「先征得同意再调用」主要寄托在 System Prompt 上（Prompt 可辅助，产品闸在 UI）。**其他工具**在 v1 **不需要**事前确认。**Agent 收束**依赖工具循环自然停止：模型不再调用工具、仅输出文本回复时，本轮自然结束，无需专用收束工具（曾计划的 `task_complete` 已废弃，见 R13）。任一工具失败时，**错误信息回传给模型**，UI **提示该次调用失败**，由 Agent **自主决定**后续动作（重试、改参、换路径或向用户说明）。**用户中断**：见 **R19**——任意时刻用户可终止当前轮次，等待中的确认按「未批准」处理，正在跑的工具按工具失败处理。
    - **Outcome：** 行为边界清晰：生图为 **先确认、后调用** 的 **UI 闸**（拒绝=「用户未批准」工具结果）；失败对用户可见且由 Agent 接续处理；用户可在任意时刻按 R19 中断。
    - **Covered by：** R14, R15, R16, R19

## Requirements

**产品与边界**

- R1. 产品为**玩具 / 实验性质**，不以上线生产、多租户隔离或合规审计为前置目标。
- R2. 支持 **多 Model**：每条记录独立配置 **baseURL、密钥、请求形态相关选项**；**不**将「单一环境变量承载全局 API Key」作为产品设计前提（进程环境仅可有与具体厂商无关的配置，如数据库路径等）。

**对话与交互**

- R3. 对话页至少提供用户可切换项：**当前 LLM**、**主生图模型**、**次生图模型**；切换主体为 **A1**，非 Agent 自动换厂商。**每条新对话创建时，上述三项选择器初始均为未选择**；用户须至少选定 **LLM** 后才允许发送消息或启动 Agent 请求（保持「通用 Agent 能力」与玩具级简化的平衡）。**主生图为可选项**：未选定主生图时，**不向 Agent 暴露任何生图工具**；当用户表达生图意图时，Agent 须通过**文本回复**告知能力缺失并提示用户先配置或选定主生图 Model（与 R9 的工具暴露规则、R14 的意图内化一致；不在产品层做硬闸阻塞发送）。**次生图为可选项**：未选定次生图时，不启用次生图工具（见 R9）。用户**回到某条已有对话**时，须**恢复该会话上次持久化**的上述三项选择（含主 / 次生图未选中的空态）。
- R4. **分辨率**作为生图侧 **自选项**；可选值必须来自 **当前选中的生图 Model 所声明的支持范围**（遵循模型与厂家真实限制），而非应用层任意硬编码列表（除非某适配层将厂家约束写死在 Model 配置中）。**控件位置（已定稿）：** 用户在 **选择主生图 / 次生图模型的同一区域或流程** 中调整分辨率等生图参数即可；**不**要求必须在对话顶栏单独再放一块分辨率控件。线框稿可不展开该控件细节。
- R5. **不提供**用户可调的「参考图张数上限」配置项；实际上限为 **该模型 / API 的最大能力**（在 Model 能力元数据中体现并在调用前校验）。**超出上限时**：UI **在调用前阻塞并提示用户调整**（如减少参考图张数），不静默截断或仅在生图失败后告知。
- R6. 对话界面**可**展示 **上下文用量可视化**：用量为**会话内累计**——每次 LLM 请求**成功返回**后，将响应中的 **usage**（如 total 或 prompt + completion，**以厂商字段为准**）**累加**，与 R10 配置的上下文窗口上限对比；**不做**本地 token 估算。**若某次响应缺少 usage、或尚无可展示的累计值，可不展示用量条**（或等价隐藏），不强行估算、不展示虚假比例。**占位（已定稿）：** 在尚无可信累计时，作曲器周边**允许**保留 **中性占位**（例如环或条显示 **0%**），且 **悬停不出现**暗示「真实上下文用量」的提示；一旦存在可信累计值，须改为仅反映真实比例，占位不得冒充真实数据。
- R7. **Compact（超上下文压缩）**：**当前版本不做**；作为后续能力在产品路线中保留一句说明即可（见 Scope Boundaries）。**撞墙行为：** 超窗口请求由 LLM API 直接返回错误，错误信息按 **R16** 回传给 Agent；由 Agent 通过文本告知用户「上下文已满，建议开新对话」。**不**做本地裁剪、滑动窗口或自动摘要。

**Agent 与生图工具**

- R8. Agent 形态需支持**多步**与**工具调用**，默认任务偏向图像，但保持通用 Agent 能力；工具集合包含 **web-search**、**image-search**、**web-fetch**（见 R17、R12）及图像生成相关能力。**对话 UI 展示名（已定稿）：** 时间线中的工具名称 **可直接使用** 与实现/协议一致的 **内部 tool 名**（可与上述注册名同一字符串），**不**强制另设仅用于界面的第二套译名。
- R9. 对 LLM 暴露的生图工具与 **主 / 次生图** 选型对齐：当用户**已配置主生图且已配置次生图**时，暴露 **两个**生图工具（主、次），各自 **input schema 随绑定 Model 的能力** 变化（例如是否包含 `negativePrompt`、参考图字段等）。当用户**仅配置主生图**时，**仅暴露主生图工具**。当用户**未配置主生图**时，**不暴露任何生图工具**（见 R3 的文本告知规则）。**主与次生图的产品语义**：两者为**收费 / 能力不同的两条独立候选 Model**，由 Agent 根据任务自由选择何时调用主、何时调用次（**不**规定 fallback、草稿优先等固定语义；调度策略由 Agent 内化）。
- R10. LLM Model 配置项中需包含 **上下文窗口大小（用于可视化对比）**，与 R6 配合。

**Agent 工作流与行为（产品级）**

- R14. 具备 **用户意图识别** 能力：至少区分 **闲聊**、**讨论（仅交换信息与观点）**、**用户要求开始动手执行**、**用户分步引导生成** 等模式，并据此调整回复与工具使用策略。**不要求**实现单独的代码层意图分类/路由模块；由 **大模型在会话与工作过程中内化** 即可，不将「意图管线」列为必须工程交付物。
- R15. **生图人工确认（产品闸，UI 驱动）：** 对 **生图模型相关工具**（主或次生图），在 **对用户展示确认并得到允许之前**，**不得**调用真实生图 API。UI 以简明文案确认即可，例如 **「是否允许 …」** 加一句用户可理解的行动摘要（**…** 由实现填入，如简要描述将进行的生图意图）。用户 **确认** 后，**再**执行生图请求并将 **成功结果** 作为工具返回值进入对话；用户 **拒绝** 时，**不执行**生图 API，并将 **「用户未批准本次工具调用」** 语义（含可选的自由文本理由）作为该次工具调用的 **返回内容** 回传给 Agent。**不设**确认超时或自动代用户决策。**确认粒度（已定稿）：** 一次**生图工具调用**对应一次确认；**与该次调用产出的图像数量、API 一次返回多张与否无关**——即「确认对象是工具调用本身，不是单张图」。Agent 若需在一轮内画多张图分多次工具调用，则对应多次独立确认。**不要求**将「须先征得用户同意才能调用生图工具」主要写进 System Prompt 来实现（Prompt 可辅助，但以 **UI 闸 + 工具返回语义** 为准）。**UI 形态（已定稿）：** 确认 **内联**在 **该次生图工具调用块**内（与工具参数、状态同一视觉上下文），**默认不采用**居中或全屏 **模态弹窗**，以免打断时间线阅读；按钮形态为 **「确认」 / 「拒绝」二元**（不区分「拒绝」与「跳过」语义，统一归并为「未批准」回传）。**其他工具**在 v1 **不需要**同类确认（除非后续产品另行扩展）。
- R16. **工具失败：** 将 **错误信息返回给 Agent**（进入模型可见的上下文中）；**UI** 须能提示 **当次工具调用失败**；由 Agent **自主决策**后续步骤（如重试、修改方案、向用户澄清或中止）；不在需求层规定固定重试次数（若后续产品需要可再补充）。

**说明：** 上述条款描述 **产品行为与约束**。**System Prompt** 的具体措辞与分节随实现迭代更新。**推荐性工作流与能力要点** 已记录在 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)，与 R8–R12, R14–R19 及 F3 对照使用，不替代本文 R 条。R15 的确认 **以 UI 与工具协议为主**，不依赖 Prompt 单独扛闸。**线框稿（如 Pencil）** 中的字体仅便于排版，**不构成**产品最终字体规范；成稿以实现侧 **`docs/design-language.md`** 为准。

**数据与运行时**

- R11. 使用 **持久化存储** 保存 Model 配置、对话与消息（及与图像引用相关的数据）；**ORM 选型为 Prisma**（用户既定决策）。
- R12. 对受 **CORS** 或密钥暴露限制的能力（如任意 URL **web-fetch**），通过 **服务端路由 / BFF** 执行，而非假定纯浏览器可完成。

- R13. **⟨已废弃⟩** 历史版本曾计划「任务完成（`task_complete`）」类专用收束工具；本版本**不再保留**该能力，Agent 收束完全由「工具循环自然停止」承担（见 F3 与 Key Decisions「Agent 收束」）。保留 R13 编号占位以避免下游派生文档（playbook、设计计划等）引用错位；下游若仍出现 R13 引用，按本条视为已废弃。
- R17. **搜索与抓取（v1）：** 须实现 **web-search**、**image-search**、**web-fetch** 三类工具（均由服务端执行，见 R12）。采用 **可插拔的多 Search Model** 架构；**web-search** 与 **image-search** 可绑定**不同** Model 记录。当前迭代默认接入 **Brave Search**；后续可增其他 Search Model，不改变「按类型选 Model」的产品模型。**Search Model 配置 schema：** 与 LLM / 生图共用同一「Model」抽象（统一设置页与持久化结构），但**字段按 type 分支**——例如 Brave Search 不沿用 `baseURL` 概念，使用专属字段。**Brave 密钥归属：** v1 默认接入的 Brave Search 密钥**必须**作为 Search Model record 在设置页配置；**不允许**通过全局环境变量 fallback（与 R2「不依赖单一全局 API Key」一致）。**web-fetch** 不绑定 Search Model（无凭证、纯 fetch 代理），不进入 Search Model 选型。
- R18. **对话内图像引用与存储：** 须具备稳定 **ID**（或等价寻址）以便 Agent 与多模态 LLM 引用。**用户上传及生图 API 返回的图像**按实现持久化（例如本地文件 + DB 元数据，规划阶段定路径与命名）。**来自搜索等、仅提供 URL 的图像**允许**只存 URL**、**不要求**下载落盘；若某步工具链或生图 API 需要图像字节，由服务端在调用时再抓取或转换。**对话删除时：** 用户上传与生图产出的本地图像随对话**级联删除**；URL-only 来源不下载、删除时无额外清理动作（URL 自然失效不影响其他对话）。

- R19. **用户中断（送达 / 停止）：** 用户可在任意时刻**中断**当前 Agent 轮次。中断时：
    - 若有正在等待 R15 确认的生图工具调用 → 视作 **「用户未批准」** 回传 Agent（与拒绝同义）；
    - 若有正在执行的工具调用 → 视作 **工具失败**（错误原因「用户中断」）按 **R16** 回传 Agent；
    - Agent 据此决定后续行为（停止本轮 / 向用户说明 / 等待用户下一条消息）。
    
    **UI 形态建议（非硬性，但作为推荐形态写入需求）：** 输入区主操作按钮按状态切换——空闲时为 **「发送」**，Agent 跑工具循环或等待 R15 确认时切换为 **「停止」**；点击「停止」即触发本条中断。其他实现形态（如独立 stop 按钮、键盘快捷键）允许，但产品上须始终存在「让用户随时停下来」的可达入口。

## Acceptance Examples

- AE1. **Covers R2, R3, F1.** 用户已配置两条生图 Model（不同 baseURL 与 Key）。在对话页将主生图从 A 切到 B，发送同一句提示；服务端请求发往 B 的 endpoint，且无需依赖全局单一 `API_KEY` 环境变量。
- AE2. **Covers R4, R5, F2.** 用户选择某主生图 Model，其元数据声明仅支持两种分辨率；分辨率下拉仅显示这两项；用户附加参考图至模型允许的最大张数，无需在 UI 配置「张数上限」字段。
- AE3. **Covers R6, R10.** 用户为当前 LLM 配置了上下文窗口 N；会话中多次成功响应均含 usage；界面将 **usage 会话累计** 相对 N 可视化。若多次无 usage，界面可不展示用量条。
- AE4. **Covers R8, R9.** 用户固定主、次生图后，Agent 在一次任务中可先调用搜索类工具，再调用主或次生图工具；用户能在对话流中看到工具名与结果（含图像）。
- AE5. **Covers R15.** Agent 拟执行主或次生图时，UI 以「是否允许 …」类文案确认（内联于该工具块，按钮为「确认 / 拒绝」二元，见 R15）；**在用户确认前**不调用生图 API；用户 **拒绝** 时该次工具返回「用户未批准」语义、**不**调用生图 API；用户 **确认** 后请求发往生图 API 并将结果作为工具返回。该次工具调用产出多张图（API 一次返回多张）时，**仍只对应一次确认**——粒度对齐到「工具调用」而非「单张图」。
- AE6. **Covers R16.** 某非生图工具调用失败，UI 显示该次失败；Agent 在后续回复中基于错误信息选择重试、改参或向用户说明，而非静默吞掉。
- AE7. **Covers R3, R17.** 用户为新对话选定 LLM 与主生图后切换至其他对话再返回，三项选择仍与离开前一致；web-search / image-search 经已配置的 Search Model（如 Brave）在服务端执行，且两类搜索可指向不同 Model；Brave 密钥取自 Search Model record，**不**接受全局环境变量 fallback。

- AE8. **Covers R3, R9, R14.** 用户**仅选定 LLM、未配置任何生图 Model**，发送「帮我画一只猫」的消息：服务端发起请求时不向 Agent 暴露任何生图工具；Agent 通过文本回复说明「当前未配置主生图 Model，请先在设置页配置或在对话顶部选择主生图 Model」，**不**触发任何生图 API 调用、**不**伪装成已生图。

- AE9. **Covers R6, R7, R10.** 用户为当前 LLM 配置上下文窗口 200k；首条消息成功响应含 usage（如 prompt + completion ≈ 5k），用量条显示约 2.5%；持续累计直到某次请求超出窗口，LLM API 直接报错；错误经 R16 回传 Agent，Agent 通过文本回复告知用户「上下文已满，建议开新对话」，**不**做本地裁剪或滑动窗口。

- AE10. **Covers R12, R17, R18.** 用户在对话中粘贴外部图片 URL；Agent 决定调用 web-fetch 抓取以分析；请求**经服务端**完成（浏览器无 CORS 报错），抓取结果（图像字节）作为工具返回值进入对话；同对话中后续生图工具按需引用该图像。该 URL 来源图**不强制落盘**；若用户后续删除整条对话，URL-only 类来源不需要额外清理动作。

- AE11. **Covers R19, R15, R16.** Agent 处于工具循环中（如连续执行 web-search → image-search → image.generate 链路）；用户点击发送区按钮（此时形态为「停止」）：
    - 若 stop 触发时正显示某次 R15 确认卡，则该确认视作 **「用户未批准」** 回传，Agent 不再调用该次生图 API；
    - 若 stop 触发时某工具正在执行，则该工具按 **工具失败（原因「用户中断」）** 回传，Agent 据此输出「已停止」类文本回复结束本轮；
    - 中断后用户仍可继续在同一对话发送下一条消息，对话历史完整保留。

- 用户能在**同一产品**内管理多家厂商的 LLM 与生图配置，并在对话页**显式切换**当前使用的 LLM 与主/次生图，且行为与所选 Model 一致；**仅选定 LLM** 即可进入对话，未配置生图时 Agent 通过文本告知能力缺失。

## Key Decisions

- **运行时与框架：** Next.js（App Router）+ **Vercel AI SDK** 作为对话与工具编排主链；**Bun** 作为包管理与本地 dev 的默认方式。
- **数据：** SQLite（或等价本地文件库）+ **Prisma**；Model 配置、密钥、对话与消息等入库。
- **密钥与 Endpoint：** 按 Model 存 **baseURL + Key（及必要头）**；不设「全应用唯一 API Key 环境变量」的产品前提。
- **用量：** 仅用请求返回的 **usage** 做**会话累计**并更新可视化；无可用数据时可不展示，或采用 **中性占位**（见 R6）；不做 token 估算。
- **Compact：** 不做（v1）。
- **新对话默认选择：** 每条新对话三项选择器初始为空；**须先选定 LLM 方可对话**；**主、次生图均为可选**——未选主生图则无任何生图 tool 暴露给 Agent，未选次生图则无次生图 tool。
- **会话选择持久化：** 回到已有对话时恢复该会话上次保存的 LLM / 主生图 / 次生图选择（含未选中的空态）。
- **生图工具暴露规则：** 主未配置 → 不暴露任何生图工具，Agent 文本告知用户能力缺失；主已配置且次未配置 → 仅暴露主生图工具；主、次均已配置 → 暴露主、次两个生图工具。
- **主 / 次生图语义：** 两者为**收费 / 能力不同的两条独立候选 Model**，由 Agent 根据任务自由选择何时调用主、何时调用次；**不**规定 fallback、草稿优先等固定语义（避免产品层把 Agent 决策权写死）。
- **搜索与抓取：** v1 含 **web-search、image-search、web-fetch**；**多 Search Model** 架构，**web-search 与 image-search 可不同 Model**；当前默认 **Brave Search**。
- **Search Model schema：** 与 LLM / 生图共用同一「Model」抽象，但**字段按 type 分支**（如 Brave 不沿用 baseURL 概念）；**Brave 密钥强制**作为 Search Model record 在设置页配置，不允许全局环境变量 fallback。
- **图像存储：** 用户上传与生图落盘按实现；**搜索等来源仅 URL 的图可只存 URL**，不必落盘；**对话删除时**本地图随之级联清理，URL-only 来源不动。
- **密钥持久化：** v1 接受密钥以**明文**形式存储于本地 SQLite（玩具 / 单机自用，与 R1 一致）；加密 / Keychain 集成 deferred。
- **Agent 收束：** 依赖工具循环自然停止——模型不再调用工具、仅输出文本时本轮结束；**无专用收束工具**（曾计划的 `task_complete` 已废弃，见 R13 占位）。意图由模型内化（**无**单独意图代码模块）。
- **Agent 生图确认：** 生图须先 UI 确认再调真实 API，确认 **内联**于工具块（见 R15），无超时，按钮 **「确认 / 拒绝」二元**（统一归并「跳过」语义为「未批准」），**确认粒度=一次工具调用**；工具错误回传并由 Agent 自主续处理。
- **用户中断（R19）：** 用户可在任意时刻中断当前轮次；UI 推荐形态为**发送 / 停止同按钮**按状态切换；中断时等待中的确认按「未批准」处理，正在跑的工具按工具失败处理。
- **工具名在 UI：** 时间线 **直接**用内部 tool 名（见 R8）。
- **分辨率：** 在 **选择生图模型** 流程中配置（见 R4）。
- **单条 Model = 单一类型：** 一条 Model 记录绑定 **一种** 类型（LLM / 生图 / Search）；同一物理模型若提供多种能力，由用户配置成多条 Model 记录，避免类型与能力元数据互相污染。
- **术语：「Model」取代「Provider」**：本项目中的配置单元称为 **Model**，代表一个具体的模型实例（如 `kimi-k2.5`、`seedream-4.5`），而非 AI SDK 意义上的 provider（服务商）。一条 Model 记录持有该模型的 endpoint、密钥与能力元数据；底层使用哪个 AI SDK 适配器为实现细节，不体现在产品层命名中。文档历史遗留的「Provider」用语与「Model」同义，以本条为准。
- **LLM 调用方式：** UI 提供内置 provider 类型选项（如 `openai-compatible`、`moonshot`、`alibaba` 等主流厂商），用户选择后服务端使用对应 AI SDK provider 实例；未被内置覆盖的厂商统一 fallback 到 `openai-compatible`。具体枚举值为工程实现细节。
- **图像生成调用方式：** 图像生成工具**不经过 AI SDK 的图像生成抽象**，直接通过 HTTP 调用各厂商图像 API（自行封装请求/响应），以支持参考图、负向提示等 provider 特定能力（AI SDK 图像生成接口不支持此类扩展参数）。

## Dependencies / Assumptions

- 所选 LLM 接口在多数情况下返回 **usage**；若某 Model 常缺失，用量可视化可不展示（见 R6）。
- **LLM Vision 假设：** v1 默认假定用户配置的 LLM Model **支持多模态 / Vision**（用于 Agent 阅读对话中已存在的图像、自检生图结果等）；若用户配置了不支持 Vision 的 LLM 又触发了多模态任务，行为由 R16 工具失败兜底（错误回传 → Agent 文本说明）——产品层**不**做 Agent 自动切换 Model、**不**做 UI 强制按 Vision 能力过滤候选 Model。
- **Brave Search** 作为 v1 默认 Search Model；密钥由用户在设置页作为 Search Model record 配置，按 Brave 产品条款使用（见 R17）。
- 生图工具通过服务端直接 HTTP 调用各厂商生图 API（不经 AI SDK 图像生成抽象）；各家差异由工具内封装处理。
- 仓库已存在 Prisma 与 Next 依赖（以当前 `package.json` 为准）。

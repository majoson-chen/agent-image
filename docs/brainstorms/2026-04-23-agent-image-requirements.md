# agent-image：对话式图像 Agent（需求说明）

## Problem Frame

单机自用的**玩具级**产品：用 **Next.js 全栈壳 + 对话界面** 驱动「类 Cursor / Claude Code」的多轮 Agent 体验，**默认任务偏向图像生成**（及与生成强相关的编排；**专业「编辑」能力以 Scope 中 v1 范围为准**），但 Agent 本身保持通用能力（工具、可见的中间步骤）。用户通过**主观体验**判断是否好用；**不做**科学对照实验或严谨效果证明。

核心矛盾：多厂商 LLM / 生图 API 的 **baseURL、鉴权、参数形态、能力上限** 各不相同；系统必须支持**多条 Provider 配置**，且**不能**依赖「整个应用只有一个环境变量 API Key」的模型。

**相关文档（推荐性，非硬性 R 条）：** Agent 默认工作流与能力告知要点见 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)，与本文 R8–R16 及 F3 对齐，供 Prompt / 实现对照。

---

## Actors

- A1. **终端用户（本人）**：配置 Provider、在对话页选择当前会话使用的模型、与 Agent 多轮交互、查看工具与上下文用量。
- A2. **Agent（服务端编排的 LLM）**：在**用户已选定**的 LLM 与主/次生图 Provider 约束下，选择工具、产生可见推理/工具步骤；**不负责**切换厂商或替换用户选定的 Provider。

---

## Key Flows

- F1. **配置 Provider**
    - **Trigger：** 用户打开 Provider / 设置相关界面。
    - **Actors：** A1
    - **Steps：** 新增或编辑一条 Provider；填写类型（**LLM** / **生图** / **Search** 等）、厂商逻辑或适配标识、**baseURL**、**API Key**（及所需附加头字段若适用）、**上下文窗口上限**（仅 LLM，用于用量可视化）、生图侧 **支持的输出分辨率/比例枚举**（来自模型或厂家真实能力）、其他能力元数据（如是否支持负向提示、参考图等，供工具 schema 与校验使用）。**Search 类**按 **R17** 配置：可插拔多 Search Provider，且 **web-search** 与 **image-search** 可绑定**不同** Provider。
    - **Outcome：** 持久化存储中存在可用的 Provider 记录；**不存在**「全局唯一 `API_KEY` 环境变量」作为产品前提。
    - **Covered by：** R2, R3, R4, R10, R17

- F2. **对话中生成与工具可见性**
    - **Trigger：** 用户在对话页发送消息；当前会话已选择 LLM、主生图、次生图（若适用）。
    - **Actors：** A1, A2
    - **Steps：** 请求经服务端调用选定 LLM；模型可调用通用工具（如联网搜索、抓取、以图搜图等，具体清单可迭代）及 **主/次** 生图工具；工具执行结果与中间产物在对话中**可读**；生图前可有搜索、分析、改写等步骤，**不把生图当作单一黑盒一次调用**。
    - **Outcome：** 用户能看到 Agent 做了什么；对话中可出现多张图（检索、中间稿、成品等），且 Agent 能**引用本轮对话内已出现的图像**（存储与 URL 规则见 **R18**；多模态序列化细节可在规划中细化）。
    - **Covered by：** R1, R5, R6, R7, R8, R9, R13–R18

- F3. **Agent 收束、意图与失败恢复（产品行为）**
    - **Trigger：** 多轮对话中模型正在执行任务或工具返回结果/错误。
    - **Actors：** A1, A2
    - **Steps：** Agent 根据对用户消息的理解区分 **闲聊 / 讨论 / 动手执行 / 分步引导** 等意图并调整策略（**不要求**单独代码层意图模块，见 R14）。**生图类工具**：模型可产生拟执行生图的工具意图；**须先经 UI 确认，再调用真实生图 API**（见 R15）。**不设**确认超时；用户仅 **允许** 或 **拒绝/跳过**。**拒绝/跳过** 作为该次工具调用的 **返回内容** 回传给模型，**不**执行生图。**不要求**将「先征得同意再调用」主要寄托在 System Prompt 上（Prompt 可辅助，产品闸在 UI）。**其他工具**在 v1 **不需要**事前确认。Agent 可在认为**当次任务已达成**时调用 **`task_complete`**（见 R13）。任一工具失败时，**错误信息回传给模型**，UI **提示该次调用失败**，由 Agent **自主决定**后续动作（重试、改参、换路径或向用户说明）。
    - **Outcome：** 行为边界清晰：生图为 **先确认、后调用** 的 **UI 闸**（拒绝/跳过=工具结果）；完成态可显式声明；失败对用户可见且由 Agent 接续处理。
    - **Covered by：** R13, R14, R15, R16

## Requirements

**产品与边界**

- R1. 产品为**玩具 / 实验性质**，不以上线生产、多租户隔离或合规审计为前置目标。
- R2. 支持 **多 Provider**：每条记录独立配置 **baseURL、密钥、请求形态相关选项**；**不**将「单一环境变量承载全局 API Key」作为产品设计前提（进程环境仅可有与具体厂商无关的配置，如数据库路径等）。

**对话与交互**

- R3. 对话页至少提供用户可切换项：**当前 LLM**、**主生图模型**、**次生图模型**；切换主体为 **A1**，非 Agent 自动换厂商。**每条新对话创建时，上述三项选择器初始均为未选择**；用户须至少选定 **LLM** 与 **主生图** 后才允许发送消息或启动 Agent 请求。**次生图为可选项**：未选定次生图时，不启用次生图工具（见 R9）。用户**回到某条已有对话**时，须**恢复该会话上次持久化**的上述三项选择（含次生图未选中的空态）。
- R4. **分辨率**作为生图侧 **自选项**；可选值必须来自 **当前选中的生图 Provider / 模型所声明的支持范围**（遵循模型与厂家真实限制），而非应用层任意硬编码列表（除非某适配层将厂家约束写死在 Provider 配置中）。**控件位置（已定稿）：** 用户在 **选择主生图 / 次生图模型（或对应 Provider）的同一区域或流程** 中调整分辨率等生图参数即可；**不**要求必须在对话顶栏单独再放一块分辨率控件。线框稿可不展开该控件细节。
- R5. **不提供**用户可调的「参考图张数上限」配置项；实际上限为 **该模型 / API 的最大能力**（在 Provider 能力元数据中体现并在调用前校验）。
- R6. 对话界面**可**展示 **上下文用量可视化**：用量为**会话内累计**——每次 LLM 请求**成功返回**后，将响应中的 **usage**（如 total 或 prompt + completion，**以厂商字段为准**）**累加**，与 R10 配置的上下文窗口上限对比；**不做**本地 token 估算。**若某次响应缺少 usage、或尚无可展示的累计值，可不展示用量条**（或等价隐藏），不强行估算、不展示虚假比例。**占位（已定稿）：** 在尚无可信累计时，作曲器周边**允许**保留 **中性占位**（例如环或条显示 **0%**），且 **悬停不出现**暗示「真实上下文用量」的提示；一旦存在可信累计值，须改为仅反映真实比例，占位不得冒充真实数据。
- R7. **Compact（超上下文压缩）**：**当前版本不做**；作为后续能力在产品路线中保留一句说明即可（见 Scope Boundaries）。

**Agent 与生图工具**

- R8. Agent 形态需支持**多步**与**工具调用**，默认任务偏向图像，但保持通用 Agent 能力；工具集合包含 **web-search**、**image-search**、**web-fetch**（见 R17、R12）及图像生成相关能力。**对话 UI 展示名（已定稿）：** 时间线中的工具名称 **可直接使用** 与实现/协议一致的 **内部 tool 名**（可与上述注册名同一字符串），**不**强制另设仅用于界面的第二套译名。
- R9. 对 LLM 暴露的生图工具与 **主 / 次生图** 选型对齐：当用户**已配置次生图**时，暴露 **两个**生图工具（主、次），各自 **input schema 随绑定 Provider 的能力** 变化（例如是否包含 `negativePrompt`、参考图字段等）。当用户**未配置次生图**时，**仅暴露主生图工具**，不出现次生图工具。
- R10. LLM Provider 配置项中需包含 **上下文窗口大小（用于可视化对比）**，与 R6 配合。

**Agent 工作流与行为（产品级）**

- R13. 提供专用工具 **`task_complete`**（**无参数**）：当 Agent 判断 **当次任务已经完成** 时调用，语义为 **收束本轮**、**不再继续主动生成/展开**（直至用户再次发送消息）。与实现层停止条件如何映射由规划阶段定。
- R14. 具备 **用户意图识别** 能力：至少区分 **闲聊**、**讨论（仅交换信息与观点）**、**用户要求开始动手执行**、**用户分步引导生成** 等模式，并据此调整回复与工具使用策略。**不要求**实现单独的代码层意图分类/路由模块；由 **大模型在会话与工作过程中内化** 即可，不将「意图管线」列为必须工程交付物。
- R15. **生图人工确认（产品闸，UI 驱动）：** 对 **生图模型相关工具**（主或次生图），在 **对用户展示确认并得到允许之前**，**不得**调用真实生图 API。UI 以简明文案确认即可，例如 **「是否允许 …」** 加一句用户可理解的行动摘要（**…** 由实现填入，如简要描述将进行的生图意图）。用户 **允许** 后，**再**执行生图请求并将 **成功结果** 作为工具返回值进入对话；用户 **拒绝或跳过** 时，**不执行**生图 API，并将 **拒绝/跳过** 语义作为该次工具调用的 **返回内容** 回传给 Agent。**不设**确认超时或自动代用户决策。**不要求**将「须先征得用户同意才能调用生图工具」主要写进 System Prompt 来实现（Prompt 可辅助，但以 **UI 闸 + 工具返回语义** 为准）。**UI 形态（已定稿）：** 确认 **内联**在 **该次生图工具调用块**内（与工具参数、状态同一视觉上下文），**默认不采用**居中或全屏 **模态弹窗**，以免打断时间线阅读；须仍满足「允许 / 拒绝或跳过」可辨与可操作。**其他工具**在 v1 **不需要**同类确认（除非后续产品另行扩展）。
- R16. **工具失败：** 将 **错误信息返回给 Agent**（进入模型可见的上下文中）；**UI** 须能提示 **当次工具调用失败**；由 Agent **自主决策**后续步骤（如重试、修改方案、向用户澄清或中止）；不在需求层规定固定重试次数（若后续产品需要可再补充）。

**说明：** 上述条款描述 **产品行为与约束**。**System Prompt** 的具体措辞与分节随实现迭代更新。**推荐性工作流与能力要点** 已记录在 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)，与 R8–R16 及 F3 对照使用，不替代本文 R 条。R15 的确认 **以 UI 与工具协议为主**，不依赖 Prompt 单独扛闸。**线框稿（如 Pencil）** 中的字体仅便于排版，**不构成**产品最终字体规范；成稿以实现侧 **`docs/design-language.md`** 为准。

**数据与运行时**

- R11. 使用 **持久化存储** 保存 Provider 配置、对话与消息（及与图像引用相关的数据）；**ORM 选型为 Prisma**（用户既定决策）。
- R12. 对受 **CORS** 或密钥暴露限制的能力（如任意 URL **web-fetch**），通过 **服务端路由 / BFF** 执行，而非假定纯浏览器可完成。
- R17. **搜索与抓取（v1）：** 须实现 **web-search**、**image-search**、**web-fetch** 三类工具（均由服务端执行，见 R12）。采用 **可插拔的多 Search Provider** 架构；**web-search** 与 **image-search** 可绑定**不同** Provider 记录。当前迭代默认接入 **Brave Search**（密钥与配额等以实现为准）；后续可增其他 Search Provider，不改变「按类型选 Provider」的产品模型。
- R18. **对话内图像引用与存储：** 须具备稳定 **ID**（或等价寻址）以便 Agent 与多模态 LLM 引用。**用户上传及生图 API 返回的图像**按实现持久化（例如本地文件 + DB 元数据，规划阶段定路径与命名）。**来自搜索等、仅提供 URL 的图像**允许**只存 URL**、**不要求**下载落盘；若某步工具链或生图 API 需要图像字节，由服务端在调用时再抓取或转换。

## Acceptance Examples

- AE1. **Covers R2, R3, F1.** 用户已配置两条生图 Provider（不同 baseURL 与 Key）。在对话页将主生图从 A 切到 B，发送同一句提示；服务端请求发往 B 的 endpoint，且无需依赖全局单一 `API_KEY` 环境变量。
- AE2. **Covers R4, R5, F2.** 用户选择某主生图 Provider，其元数据声明仅支持两种分辨率；分辨率下拉仅显示这两项；用户附加参考图至模型允许的最大张数，无需在 UI 配置「张数上限」字段。
- AE3. **Covers R6, R10.** 用户为当前 LLM 配置了上下文窗口 N；会话中多次成功响应均含 usage；界面将 **usage 会话累计** 相对 N 可视化。若多次无 usage，界面可不展示用量条。
- AE4. **Covers R8, R9.** 用户固定主、次生图后，Agent 在一次任务中可先调用搜索类工具，再调用主或次生图工具；用户能在对话流中看到工具名与结果（含图像）。
- AE5. **Covers R15.** Agent 拟执行主或次生图时，UI 以「是否允许 …」类文案确认（**可**内联于该工具块，见 R15）；**在用户允许前**不调用生图 API；用户 **拒绝/跳过** 时该次工具返回相应语义、**不**调用生图 API；用户 **允许** 后请求发往生图 API 并将结果作为工具返回。
- AE6. **Covers R13.** 多步任务结束后 Agent 调用 **`task_complete`**（无参数）；之后本轮不再自动追加工具调用或长推理，直至用户新消息。
- AE7. **Covers R16.** 某非生图工具调用失败，UI 显示该次失败；Agent 在后续回复中基于错误信息选择重试、改参或向用户说明，而非静默吞掉。
- AE8. **Covers R3, R17.** 用户为新对话选定 LLM 与主生图后切换至其他对话再返回，三项选择仍与离开前一致；web-search / image-search 经已配置的 Search Provider（如 Brave）在服务端执行，且两类搜索可指向不同 Provider。

## Success Criteria

- 用户能在**同一产品**内管理多家厂商的 LLM 与生图配置，并在对话页**显式切换**当前使用的 LLM 与主/次生图，且行为与所选 Provider 一致。
- 生图参数（分辨率等）与参考图能力**不违背**各模型声明的上限；无「人为张数上限」配置项。
- 上下文用量条在**有可用累计 usage 时**可信地反映相对配置窗口的比例；**无 usage 或不可累计时**可不展示，或采用 **不冒充真实累计** 的中性占位（见 R6）；不造假数字。
- 需求文档足以让 `/ce-plan` **少发明**产品行为：范围、边界、Actor 分工、Agent 收束/确认/失败策略与核心流程已写明。

## Scope Boundaries

### v1 范围（交付侧重）

- **核心：** 文生图、带参考图生图（在模型/API 能力内）、主/次生图工具、R15 **先确认后调用** 的 UI 闸（**内联**于工具块，见 R15）、**web-search / image-search / web-fetch**（R17，默认 Brave 接入）、工具链可见、playbook 推荐路径及 R13–R16 行为约束；对话内图像存储规则见 **R18**。
- **图像编辑**（如独立 inpaint/局部重绘管线、精确改字等专业编辑工作流）：**不作为 v1 硬性验收项**；若实现中因 Provider 或工具自然覆盖而可用，可作为增量，**不写入**当前 R 条验收。

### Deferred for later

- **上下文 compact**：超窗口时的摘要、裁剪或滑动窗口策略；仅在文档/路线中预留，实现排期后置。
- **System Prompt 终稿**：分节结构、语气、意图策略细表及与 R13–R16 的对照文案；与产品迭代同步，不在本文件展开全文。
- **生图工作流细节**：具体工具列表、与各家 API 的一一适配，可在实现迭代中细化（R13–R16 已固定行为约束）。**推荐性**步骤与能力要点见 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)。
- **严谨效果验证**：对照实验、固定题集、自动化评测体系。

### Outside this product's identity

- 生产级多用户 SaaS、计费、密钥托管合规、企业审计。
- 「单一全局厂商 Key 走天下」作为默认架构假设。

## Key Decisions

- **运行时与框架：** Next.js（App Router）+ **Vercel AI SDK** 作为对话与工具编排主链；**Bun** 作为包管理与本地 dev 的默认方式。
- **数据：** SQLite（或等价本地文件库）+ **Prisma**；Provider、密钥、对话与消息等入库。
- **密钥与 Endpoint：** 按 Provider 存 **baseURL + Key（及必要头）**；不设「全应用唯一 API Key 环境变量」的产品前提。
- **用量：** 仅用请求返回的 **usage** 做**会话累计**并更新可视化；无可用数据时可不展示，或采用 **中性占位**（见 R6）；不做 token 估算。
- **Compact：** 不做（v1）。
- **新对话默认选择：** 每条新对话三项选择器初始为空；须先选定 **LLM + 主生图** 方可对话；**次生图可选**，未选则无次生图 tool。
- **会话选择持久化：** 回到已有对话时恢复该会话上次保存的 LLM / 主生图 / 次生图选择。
- **次生图未配置：** 仅向 LLM 暴露主生图工具，不暴露次生图工具。
- **搜索与抓取：** v1 含 **web-search、image-search、web-fetch**；**多 Search Provider** 模型，**web-search 与 image-search 可不同 Provider**；当前默认 **Brave Search**。
- **图像存储：** 用户上传与生图落盘按实现；**搜索等来源仅 URL 的图可只存 URL**，不必落盘。
- **Agent：** 工具名 **`task_complete`**、**无参数**，用于收束本轮；意图由模型内化（**无**单独意图代码模块）；**生图须先 UI 确认再调真实 API**，确认 **内联**于工具块（见 R15），无超时，允许 / 拒绝或跳过，拒绝或跳过=工具返回；工具错误回传并由 Agent 自主续处理。
- **工具名在 UI：** 时间线 **直接**用内部 tool 名（见 R8）。
- **分辨率：** 在 **选择生图模型/Provider** 流程中配置（见 R4）。

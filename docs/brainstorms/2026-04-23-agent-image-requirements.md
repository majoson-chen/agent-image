# agent-image：对话式图像 Agent（需求说明）

## Problem Frame

单机自用的**玩具级**产品：用 **Next.js 全栈壳 + 对话界面** 驱动「类 Cursor / Claude Code」的多轮 Agent 体验，**默认任务偏向图像生成与编辑**，但 Agent 本身保持通用能力（工具、可见的中间步骤）。用户通过**主观体验**判断是否好用；**不做**科学对照实验或严谨效果证明。

核心矛盾：多厂商 LLM / 生图 API 的 **baseURL、鉴权、参数形态、能力上限** 各不相同；系统必须支持**多条 Provider 配置**，且**不能**依赖「整个应用只有一个环境变量 API Key」的模型。

**相关文档（推荐性，非硬性 R 条）：** Agent 默认工作流与能力告知要点见 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)，与本文 R8–R16 及 F3 对齐，供 Prompt / 实现对照。

---

## Actors

- A1. **终端用户（本人）**：配置 Provider、在对话页选择当前会话使用的模型、与 Agent 多轮交互、查看工具与上下文用量。
- A2. **Agent（服务端编排的 LLM）**：在**用户已选定**的 LLM 与主/次生图 Provider 约束下，选择工具、产生可见推理/工具步骤；**不负责**切换厂商或替换用户选定的 Provider。

---

## Key Flows

- F1. **配置 Provider**
  - **Trigger：** 用户打开 Provider / 设置相关界面。
  - **Actors：** A1
  - **Steps：** 新增或编辑一条 Provider；填写类型（LLM / 生图等）、厂商逻辑或适配标识、**baseURL**、**API Key**（及所需附加头字段若适用）、**上下文窗口上限**（仅 LLM，用于用量可视化）、生图侧 **支持的输出分辨率/比例枚举**（来自模型或厂家真实能力）、其他能力元数据（如是否支持负向提示、参考图等，供工具 schema 与校验使用）。
  - **Outcome：** 持久化存储中存在可用的 Provider 记录；**不存在**「全局唯一 `API_KEY` 环境变量」作为产品前提。
  - **Covered by：** R2, R3, R4, R10

- F2. **对话中生成与工具可见性**
  - **Trigger：** 用户在对话页发送消息；当前会话已选择 LLM、主生图、次生图（若适用）。
  - **Actors：** A1, A2
  - **Steps：** 请求经服务端调用选定 LLM；模型可调用通用工具（如联网搜索、抓取、以图搜图等，具体清单可迭代）及 **主/次** 生图工具；工具执行结果与中间产物在对话中**可读**；生图前可有搜索、分析、改写等步骤，**不把生图当作单一黑盒一次调用**。
  - **Outcome：** 用户能看到 Agent 做了什么；对话中可出现多张图（检索、中间稿、成品等），且 Agent 能**引用本轮对话内已出现的图像**（实现方式由规划阶段定）。
  - **Covered by：** R1, R5, R6, R7, R8, R9, R13–R16

- F3. **Agent 收束、意图与失败恢复（产品行为）**
  - **Trigger：** 多轮对话中模型正在执行任务或工具返回结果/错误。
  - **Actors：** A1, A2
  - **Steps：** Agent 根据对用户消息的理解区分 **闲聊 / 讨论 / 动手执行 / 分步引导** 等意图并调整策略（**不要求**单独代码层意图模块，见 R14）。**生图类工具**：模型可照常 **发起工具调用**；**UI 在调用发生后**弹出确认；用户 **批准** 则执行真实生图并将成功结果作为工具返回，用户 **拒绝** 则将 **拒绝** 作为该次工具调用的 **返回内容** 回传给模型，由 Agent 继续对话。**不要求**将「先征得同意再调用」主要寄托在 System Prompt 上（Prompt 可辅助，产品闸在 UI）。**其他工具**在 v1 **不需要**事前确认。Agent 可在认为**当次任务已达成**时调用专用 **「任务完成」类工具**，作为**停止继续生成/收束本轮主动工作**的信号。任一工具失败时，**错误信息回传给模型**，UI **提示该次调用失败**，由 Agent **自主决定**后续动作（重试、改参、换路径或向用户说明）。
  - **Outcome：** 行为边界清晰：生图有 **UI 确认闸**（拒绝=工具结果）；完成态可显式声明；失败对用户可见且由 Agent 接续处理。
  - **Covered by：** R13, R14, R15, R16

## Requirements

**产品与边界**

- R1. 产品为**玩具 / 实验性质**，不以上线生产、多租户隔离或合规审计为前置目标。
- R2. 支持 **多 Provider**：每条记录独立配置 **baseURL、密钥、请求形态相关选项**；**不**将「单一环境变量承载全局 API Key」作为产品设计前提（进程环境仅可有与具体厂商无关的配置，如数据库路径等）。

**对话与交互**

- R3. 对话页至少提供用户可切换项：**当前 LLM**、**主生图模型**、**次生图模型**；切换主体为 **A1**，非 Agent 自动换厂商。**每条新对话创建时，上述三项选择器初始均为未选择**；用户须至少选定 **LLM** 与 **主生图** 后才允许发送消息或启动 Agent 请求。**次生图为可选项**：未选定次生图时，不启用次生图工具（见 R9）。
- R4. **分辨率**作为生图侧 **自选项**；可选值必须来自 **当前选中的生图 Provider / 模型所声明的支持范围**（遵循模型与厂家真实限制），而非应用层任意硬编码列表（除非某适配层将厂家约束写死在 Provider 配置中）。
- R5. **不提供**用户可调的「参考图张数上限」配置项；实际上限为 **该模型 / API 的最大能力**（在 Provider 能力元数据中体现并在调用前校验）。
- R6. 对话界面展示 **上下文用量可视化**（例如已用相对配置的上限）；用量数据**仅**在每次 LLM 请求**成功返回后**，根据响应中的 **usage（如 total / prompt + completion，以厂商字段为准）** 更新；**不做**本地 token 估算。若某次响应缺少 usage，界面需降级展示（如显示「不可用」或沿用上一轮），不强行估算。
- R7. **Compact（超上下文压缩）**：**当前版本不做**；作为后续能力在产品路线中保留一句说明即可（见 Scope Boundaries）。

**Agent 与生图工具**

- R8. Agent 形态需支持**多步**与**工具调用**，默认任务偏向图像，但保持通用 Agent 能力；工具集合包含通用能力（搜索、抓取等）与图像相关能力。
- R9. 对 LLM 暴露的生图工具与 **主 / 次生图** 选型对齐：当用户**已配置次生图**时，暴露 **两个**生图工具（主、次），各自 **input schema 随绑定 Provider 的能力** 变化（例如是否包含 `negativePrompt`、参考图字段等）。当用户**未配置次生图**时，**仅暴露主生图工具**，不出现次生图工具。
- R10. LLM Provider 配置项中需包含 **上下文窗口大小（用于可视化对比）**，与 R6 配合。

**Agent 工作流与行为（产品级）**

- R13. 提供专用 **「任务完成」类工具**（名称与参数由实现定义）：当 Agent 判断 **当次任务已经完成** 时，应可调用该工具，语义为 **收束本轮**、**不再继续主动生成/展开**（直至用户再次发送消息）。具体与实现层停止条件如何映射由规划阶段定。
- R14. 具备 **用户意图识别** 能力：至少区分 **闲聊**、**讨论（仅交换信息与观点）**、**用户要求开始动手执行**、**用户分步引导生成** 等模式，并据此调整回复与工具使用策略。**不要求**实现单独的代码层意图分类/路由模块；由 **大模型在会话与工作过程中内化** 即可，不将「意图管线」列为必须工程交付物。
- R15. **生图人工确认（产品闸，UI 驱动）：** 对 **生图模型相关工具**（主或次生图），在 Agent **已发起该工具调用**之后，由 **UI** 向用户展示确认（展示内容由实现定义，例如拟执行参数摘要）。用户 **批准** 后，再执行真实生图请求并将 **成功结果** 作为工具返回值进入对话；用户 **拒绝** 时，**不执行**生图 API，并将 **用户拒绝** 作为该次工具调用的 **返回内容** 回传给 Agent，由其自主续写。**不要求**将「须先征得用户同意才能调用生图工具」主要写进 System Prompt 来实现（Prompt 可辅助，但以 **UI 拦截 + 工具返回语义** 为准）。**其他工具**在 v1 **不需要**同类确认（除非后续产品另行扩展）。
- R16. **工具失败：** 将 **错误信息返回给 Agent**（进入模型可见的上下文中）；**UI** 须能提示 **当次工具调用失败**；由 Agent **自主决策**后续步骤（如重试、修改方案、向用户澄清或中止）；不在需求层规定固定重试次数（若后续产品需要可再补充）。

**说明：** 上述条款描述 **产品行为与约束**。**System Prompt 的具体措辞、分节与可选指导流** 另行讨论与迭代，不阻塞本文件作为需求基线。R15 的确认 **以 UI 与工具协议为主**，不依赖 Prompt 单独扛闸。

**数据与运行时**

- R11. 使用 **持久化存储** 保存 Provider 配置、对话与消息（及与图像引用相关的数据）；**ORM 选型为 Prisma**（用户既定决策）。
- R12. 对受 **CORS** 或密钥暴露限制的能力（如任意 URL **web-fetch**），通过 **服务端路由 / BFF** 执行，而非假定纯浏览器可完成。

## Acceptance Examples

- AE1. **Covers R2, R3, F1.** 用户已配置两条生图 Provider（不同 baseURL 与 Key）。在对话页将主生图从 A 切到 B，发送同一句提示；服务端请求发往 B 的 endpoint，且无需依赖全局单一 `API_KEY` 环境变量。
- AE2. **Covers R4, R5, F2.** 用户选择某主生图 Provider，其元数据声明仅支持两种分辨率；分辨率下拉仅显示这两项；用户附加参考图至模型允许的最大张数，无需在 UI 配置「张数上限」字段。
- AE3. **Covers R6, R10.** 用户为当前 LLM 配置了上下文窗口 N；一轮对话结束后响应含 usage；界面显示本轮累计或会话用量相对 N 的可视化（具体展示口径由规划定义，但必须基于返回的 usage，非估算）。
- AE4. **Covers R8, R9.** 用户固定主、次生图后，Agent 在一次任务中可先调用搜索类工具，再调用主或次生图工具；用户能在对话流中看到工具名与结果（含图像）。
- AE5. **Covers R15.** Agent 已发起主或次生图工具调用后，UI 弹出确认；用户 **拒绝** 时该次工具返回内容为拒绝语义、**不**调用生图 API；用户 **批准** 后请求发往生图 API 并将结果作为工具返回。
- AE6. **Covers R13.** 多步任务结束后 Agent 调用「任务完成」类工具；之后本轮不再自动追加工具调用或长推理，直至用户新消息。
- AE7. **Covers R16.** 某非生图工具调用失败，UI 显示该次失败；Agent 在后续回复中基于错误信息选择重试、改参或向用户说明，而非静默吞掉。

## Success Criteria

- 用户能在**同一产品**内管理多家厂商的 LLM 与生图配置，并在对话页**显式切换**当前使用的 LLM 与主/次生图，且行为与所选 Provider 一致。
- 生图参数（分辨率等）与参考图能力**不违背**各模型声明的上限；无「人为张数上限」配置项。
- 上下文用量条在**有 usage 时**可信地反映接口返回值相对配置窗口的比例；无 usage 时有明确降级而非假数字。
- 需求文档足以让 `/ce-plan` **少发明**产品行为：范围、边界、Actor 分工、Agent 收束/确认/失败策略与核心流程已写明。

## Scope Boundaries

### Deferred for later

- **上下文 compact**：超窗口时的摘要、裁剪或滑动窗口策略；仅在文档/路线中预留，实现排期后置。
- **System Prompt 终稿**：分节结构、语气、意图策略细表及与 R13–R16 的对照文案；与产品迭代同步，不在本文件展开全文。
- **生图工作流细节**：具体工具列表、与各家 API 的一一适配，可在实现迭代中细化（R13–R16 已固定行为约束）。**推荐性**步骤与能力要点见 [`2026-04-23-agent-image-agent-playbook.md`](2026-04-23-agent-image-agent-playbook.md)。
- **严谨效果验证**：对照实验、固定题集、自动化评测体系。

### Outside this product's identity

- 生产级多用户 SaaS、计费、密钥托管合规、企业审计。
- 「单一全局厂商 Key 走天下」作为默认架构假设。

## Key Decisions

- **运行时与框架：** Next.js（App Router）+ **Vercel AI SDK** 作为对话与工具编排主链；**Bun** 作为包管理与本地 dev 的默认方式。
- **数据：** SQLite（或等价本地文件库）+ **Prisma**；Provider、密钥、对话与消息等入库。
- **密钥与 Endpoint：** 按 Provider 存 **baseURL + Key（及必要头）**；不设「全应用唯一 API Key 环境变量」的产品前提。
- **用量：** 仅用请求返回的 **usage** 更新可视化；不做 token 估算。
- **Compact：** 不做（v1）。
- **新对话默认选择：** 每条新对话三项选择器初始为空；须先选定 **LLM + 主生图** 方可对话；**次生图可选**，未选则无次生图 tool。
- **次生图未配置：** 仅向 LLM 暴露主生图工具，不暴露次生图工具。
- **Agent：** 「任务完成」工具收束；意图由模型内化（**无**单独意图代码模块）；**生图工具经 UI 事后确认**，拒绝=工具返回；工具错误回传并由 Agent 自主续处理。

## Dependencies / Assumptions

- 所选 LLM 接口在多数情况下返回 **usage**；若某 Provider 常缺失，由实现阶段决定展示策略（见 R6）。
- 生图与各厂商 API 的差异通过 **Provider 元数据 + 适配层**（含可选 AI SDK custom provider）吸收；长尾可走 tool 内 HTTP。
- 仓库已存在 Prisma 与 Next 依赖（以当前 `package.json` 为准）。

## Outstanding Questions

### Resolve Before Planning

- （暂无 —— 已确认：新对话初始为空须选齐 LLM+主生图；未配置次生图时仅主生图 tool。）

### Deferred to Planning

- **[Technical]** 对话内图像引用的稳定 ID、存储形态（DB blob vs 文件路径 vs 外链）及传入多模态 LLM 的序列化方式。
- **[Technical]** 通用工具（web-search / image-search / web-fetch）的首批实现范围与第三方 API 选型。
- **[Product/Prompt]** 「任务完成」工具的精确命名、参数及与停止生成的边界（例如与闲聊收束的区分）。**意图四类不对用户做标签化 UI 的强制要求**（由模型内化即可）。
- **[Product/UX]** 生图工具 **UI 确认** 的展示形态（摘要字段、超时/取消语义、与流式工具事件的时序）及与 AI SDK approval/自定义中间层的映射。

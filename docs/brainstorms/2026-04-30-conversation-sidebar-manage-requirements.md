# agent-image：侧栏会话重命名 / 删除 + Agent 重命名会话（需求）

与总览 [`2026-04-23-agent-image-requirements.md`](2026-04-23-agent-image-requirements.md) 相容：在用户与 Agent 共用「对话持久化」前提下，补足**会话元数据在用户侧可调**的能力；不影响既有 R8 工具命名规则（时间线仍可展示内部 tool 名）。

---

## Problem Frame

当前侧栏（`app/Sidebar.tsx`）仅能通过点击标题进入会话，**无法在列表内改名或删除**；会话标题仅能靠其它间接方式或未覆盖路径更新。

用户需要：

1. 在侧边栏对每个会话执行**重命名**、**删除**（删除需明示确认，沿用单机玩具数据模型：删会话即 cascade 会话级资源，与既有对话删除语义一致）。
2. 授权 **Agent** 在服务端根据对话内容**将当前会话改名为短标题**，作为独立工具可调，行为与手动改名结果一致。

本需求不涉及多租户、权限模型；延续 **R1 玩具属性**。

---

## Actors

- **A1. 终端用户（本人）：** 在侧栏操作系统中的会话条目（重命名、删除、仍能点击进入对话）。
- **A2. Agent（服务端编排的 LLM）：** 在满足工具暴露策略时可选调用「仅改当前会话标题」的工具；**不负责**代为删除会话。

---

## Key Flows

- **F1. 侧栏重命名**
    - **Trigger：** 用户点击某项的「重命名」。
    - **Actors：** A1
    - **Steps：** 打开轻量表单（首版：**模态对话框 + 单行输入**，见 Key Decisions）；用户输入或修改标题 → 提交 → 服务端校验并持久化 → 列表展示更新。
    - **Outcome：** 该会话的展示标题与用户输入一致（trim）；侧栏与其它入口若共享同一数据源则一并一致。
    - **Covered by：** R2, R3, R9

- **F2. 侧栏删除**
    - **Trigger：** 用户点击「删除」并确认。
    - **Actors：** A1
    - **Steps：** 二次确认 → 删除会话及相关会话级持久化资源（沿用现有服务端删除语义）→ 侧栏移除该项。
    - **Outcome：** 会话不再列出；不应残留指向已删除会话的稳定入口导致白屏。**若删除的是当前正在查看的对话，**必须先于或同时完成路由处理，使用户落在仍存在的会话或明确空路由（细节见 R4）。
    - **Covered by：** R1, R4, R10

- **F3. Agent 重命名当前对话**
    - **Trigger：** Agent 在多轮中选工具调用。
    - **Actors：** A2（发起），系统（校验与持久化）
    - **Steps：** Agent 传入**单行标题字符串**；服务端将**当前请求关联的会话**标题更新为该字符串（不写死「必须先总结再调工具」，由模型在上下文中自定）。
    - **Outcome：** 与 F1 **同一持久化语义**；用户稍后打开侧栏或刷新即见新标题；若用户在同一时段手动改名，**以后一次成功写入为准**（无 CRDT）。
    - **Covered by：** R3, R6, R7, R12

## Requirements

**侧栏与用户操作**

- R1. 每条会话在行内除「点击进入」外，提供互不遮挡的入口：**删除**（需确认）、**重命名**。交互首版可为标题链接 + 操作控件分区，避免删除/重命名与整行跳转完全重叠（见构思稿 layout F）。
- R2. **重命名**：用户提交的有效标题须为非空（trim 后）；产品规定**长度上限**（与 Agent 共用，见 Key Decisions 建议区间）；非法提交由服务端拒绝并给用户可读反馈。
- R3. **删除**：必须经过**不可逆**语义下的确认（单次确认即可；不设「撤回删除」）。
- R4. **删除当前所在会话：** 成功后用户不得停留在已无会话 ID 的对话页。**默认行为：** 跳转至仍存在的一条会话（建议列表排序中的下一条可读候选，或由实现取「最近更新时间」第一的剩余会话）；若已无会话，落到产品可接受的空路由（如不存在的 `/conversations` 列表占位或等价策略），**不得以无限 loading / 静默失败代替**。
- R5. 侧栏在数据变更后应尽快反映会话列表顺序与标题（与现有 `updatedAt` 语义一致者可沿用）。

**服务端与一致性**

- R6. 用户发起的重命名与 Agent 发起的重命名必须经过**相同的标题校验规则**写入同一字段，避免出现「一种入口允许、另一种拒绝」的分叉。
- R7. 暴露给 Agent 的改名能力**仅能作用于当前聊天会话**：工具层不得依赖模型传入任意会话 id；若未来扩展，本节 R7 须经修订。

**Agent 工具（产品语义）**

- R8. 增加工具，**内部注册名建议** `conversation-rename`（与用户可见时间线对齐 R8：直接展示内部名）。工具仅接受**标题字符串参数**（长度与 R2 一致）。
- R9. 该工具**不需要** R15 风格的「事前 UI 审批」（非真实生图/扣费类外部 API）；结果可在时间线中可见与普通工具一致。
- R10. `lib/ai/system-prompt.ts` 所构建的指令中可增加**简短、非强制**说明：若有帮助，可选用该工具为当前会话起一个简短标题；**不得**设计成每轮必调或与用户消息耦合的硬门禁。

## Acceptance Examples

- AE1. **Covers R1, R2, F1.** 用户在侧栏对会话 A 点「重命名」，在确认框中提交「周报草图」，侧栏与进入该会话后的上下文展示中，标题与该字符串一致（trim 生效）。
- AE2. **Covers R2, R3, R7, F3。** 用户仅在对话中发问，Agent 调用 `conversation-rename`，参数为「图示需求收集」；成功后侧栏会话标题变更为该字符串，且与用户稍后在侧栏手动改名使用同一校验。
- AE3. **Covers R3, R4, F2。** 用户正在会话 B 的对话页，于侧栏删除会话 B 并确认；页面导航离开已删 id，不出现针对 B 的对话区持续展示。
- AE4. **Covers R4.** 仅存一条会话时被删除确认后，不出现仍然指向旧 id 的可发送对话页。

---

## Success Criteria

- 单机用户可不离开侧栏完成对会话的常见生命周期管理：**改名**与**删除**。
- Agent 与用户改名**行为一致**，无「仅存一边」的第二套标题。
- 删除当前会话不出现「假死路由」或可复现的卡死视图。

---

## Scope Boundaries

- **Defer：** 快捷键、批量删除、回收站恢复、拖拽排序、c「仅撤销标题」撤销栈。
- **Defer：** 「Agent 口头建议标题 + 用户在独立 UI 一键接受」双线流程的首发（构思已标记为备选；本条不纳入 v1 验收）。
- **Out：** 多端同步、会话分享、服务端审计日志产品线。

---

## Key Decisions

1. **重命名 UX 首轮：** **模态 + 单行输入**，满足可达性与 DaisyUI 语义色惯例；之后再考虑行内就地编辑。
2. **标题长度：** 与用户输入及模型输出共用上限；数值由实现选定（例如 1–120 字符）并在服务端统一 enforce。
3. **冲突：** 多端仅本人；最终成功写入者优先，无合并策略。

---

## Dependencies / Assumptions

- 延续现有：**删除会话级联图像等**已实现行为不变；本需求不缩小也不扩大「删除后果」的认知（仍面向玩具自用）。
- 现有「对话加载」对已删 id 应具有 `notFound` 或与删除联动跳转；若在实现时发现缺口，归入实现计划补齐，产品期望见 R4。

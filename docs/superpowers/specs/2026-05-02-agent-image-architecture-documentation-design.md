# agent-image：架构文档双轨设计（Canvas + specs）

**状态：** 已定稿（供实现「画布改版」与后续重构对齐）  
**日期：** 2026-05-02  
**受众：** 本文件主要供 Agent / 协作检索与固化决策；**人类日常阅读入口仍为 Cursor Canvas**（`canvases/`），两套介质职责分离。

---

## 1. 目标

1. **人类**：在**不看仓库内计划 MD、不依赖通读源码**的前提下，建立对项目的**系统架构与运行模型**的理解；Canvas 是人类的**唯一**习惯性入口。
2. **协作 / Agent**：在 `docs/superpowers/specs/` 留存可追溯的设计决策与结构约定，便于检索、对计划与实现，避免口头约定漂移。

---

## 2. 双轨分工（独立、不对立）

| 轨道 | 位置 | 用途 |
|------|------|------|
| **人类线** | `canvases/*.canvas.tsx` | 叙事化、分段可读的架构模型 + 伪代码 + 短锚点表 |
| **协作线** | `docs/superpowers/specs/*.md` | 设计定稿、范围、模板、粒度规则、与迭代/重构的同步策略 |

两者**不互相替代**：spec 不追求替代 Canvas 的阅读体验；Canvas 不承载冗长的流程检查表或 Agent 专属指令。

---

## 3. 内容原则（相对「产品说明书」）

**要做（架构模型）：**

- 信任边界、请求主时序、核心**不变量**（例如续写同一条助手消息、中断语义）。
- 核心实体的**生命周期与状态迁移**（消息 / parts、工具审批、选型与绑定）。
- 模块或分层的**职责与依赖方向**（概念层）；必要时用**简化伪代码**表达控制流，避免读者扎进实现细节。
- **短锚点**：入口文件、关键类型/函数名，用于「验证理解」而非「替代正文」。

**不做或极简：**

- 逐步教用户「怎么点界面」的长说明。
- 独占篇幅的 R 条款复述（仅在解释**为何存在某不变量**时一句话带过即可）。
- 大段真实源码摘录（除非极短的签名级引用；优先伪代码）。

---

## 4. 叙事主干（用户已选）

- **混合结构（Hybrid）**：  
  - **先总览**：边界 + **单次请求主时序**。  
  - **再专题**：消息模型、工具模型、运行时/协议、模型与选型 —— **四块同等深度**，禁止「只有一块写硬、其余百科式带过」。
- **与源码的绑定方式（用户已选）**：  
  - 正文用**模型语言 + 简化伪代码**；  
  - **允许引用**具体路径/类型名作为锚点；  
  - 目标减轻读者**通读**代码的负担，而不是禁止出现代码指涉。

---

## 5. 画布落地形态（方案 1：六张不换文件名，换内在骨架）

沿用现有六个中文文件名；每张画布采用**统一四段结构**（顺序固定，便于扫读）：

1. **模型与不变量**  
   读者应能回答：「这里谈的是哪类对象/边界？在什么前提下什么一定成立/一定不成立？」

2. **时序或状态**  
   用**步骤表**或**短伪代码**（10～25 行量级上限；无则只用步骤表）描述主路径；分支只保留对架构有影响的（批准、中断、续写）。

3. **锚点表**  
   表格三列建议：`概念` \| `入口或类型（短）` \| `备注（可选）`。不加长代码块。

4. **边界与坑（可选，极简）**  
   仅列**已知**且影响架构理解的一致性或刷新问题；禁止展开成 FAQ。

### 5.1 各文件主题对焦（保持与现有一致，升级深度）

| 画布 | 对焦 |
|------|------|
| `维护者手册.canvas.tsx` | 阅读协议（四段怎么用）、常用命令、画布索引；「对外行为约束」极短，服务理解系统而非教操作 |
| `架构总览.canvas.tsx` | 信任边界 + 单次请求主轴时序 + 目录/子系统职责映射 |
| `AI运行时与聊天接口.canvas.tsx` | POST 生命周期、Agent 多步循环、流式与落库节奏、中断、续写不变量 |
| `消息与parts模型.canvas.tsx` | 实体与投影、parts 状态（含审批前后）、用户消息流入边 |
| `工具注册与审批.canvas.tsx` | 工具快照的输入依赖、输出、审批协议状态机 |
| `模型工厂与会话选型.canvas.tsx` | 配置 vs 会话选择、capabilities / binding 的单源事实、Provider 插槽 |

## 6. 伪代码粒度规则

- **目的**：表意控制流与职责边界，**非**可编译实现。
- **长度**：单画布内伪代码块总计建议 ≤ 25 行；超过则拆成第二个块或改步骤表。
- **允许**：`// 不变量：` 注释、省略错误处理分支（在「边界与坑」里点一句即可）。
- **禁止**：贴项目真实大段代码、逐 API 参数列举。

---

## 7. 与「计划全盘重构」的关系

- 本文档定义的是**人类 Canvas 的形态与信息架构**，以及 **specs 中如何固化这类决策**。
- **当代码重构落地后**：必须做一次 **Canvas 与本文对齐修订**（锚点表、时序、不变量以新实现为准）；本文的**段落模板与双轨分工**持续有效，除非另开 spec 废止。
- 重构的**产品/技术范围**若单独立项，应用**单独的 spec**（如 `...-refactor-design.md`）描述目标架构；本文件不替代该类 spec。
